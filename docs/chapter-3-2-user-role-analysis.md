## 3.2 用户角色分析

本系统采用基于角色的访问控制（RBAC）模型，结合 JWT 鉴权、后端接口授权与前端路由守卫，实现“可浏览、可操作、可管理”三层能力划分。当前系统主要定义三类角色：普通用户（ROLE_USER）、系统管理员（ROLE_ADMIN）与游客（ROLE_GUEST）。

从实现机制看，后端在安全配置中对不同 URL 与 HTTP 方法进行角色约束，前端在路由守卫中进一步进行页面级拦截与跳转，形成“后端硬约束 + 前端软引导”的双重控制策略，既保证安全性，也保证交互一致性。

### 3.2.1 普通用户

普通用户是系统的核心业务使用者，承担“宠物档案记录、回忆沉淀、社区互动”的主要功能闭环。

1. 角色定位

普通用户对应真实注册账号，具备完整的个人功能使用权，能够在系统中建立长期数字纪念内容，并参与社区交流。

2. 主要权限

- 认证与账户：注册、登录、查看个人信息、完善昵称/头像/简介。
- 宠物与回忆：创建/编辑/删除宠物信息，维护宠物回忆时间线，上传图片与视频。
- 宠物档案与提醒：记录体重、疫苗、体检、驱虫、用药等事件，设置提醒并处理“已完成/稍后”状态。
- 社区互动：发布动态、点赞、评论、关注、私信、举报、话题参与等。
- 公开分享：生成并分享宠物主页链接，供外部访问公开内容。

3. 权限边界与状态约束

- 普通用户无法访问管理员接口与管理员页面。
- 当账号被管理员冻结时，登录会被阻断。
- 当账号被设置“限制发布”时，可浏览内容但发帖与互动能力会受到约束。

4. 设计意义

普通用户权限覆盖“记录 + 互动 + 沉淀”全流程，能够满足毕业设计场景下的核心业务目标，同时通过账号状态管控机制降低滥用风险。

### 3.2.2 系统管理员

系统管理员负责平台治理与运行维护，是系统稳定性与内容安全的保障角色。

1. 角色定位

管理员属于高权限运维角色，重点工作不是日常内容生产，而是用户治理、内容审核、风险处置与系统态势监测。

2. 主要权限

- 用户账号管理：查看用户列表与账号状态，执行冻结/解封、限制发布、警告等操作。
- 社区内容审核：对帖子和评论执行屏蔽、删除、违规处理。
- 举报与申诉处理：处理用户举报单与账号申诉单，闭环异常处置流程。
- 系统概览维护：查看用户数、帖子数、评论数、多媒体存储量、关键接口状态等运行指标。
- 治理接口范围：通过 `/api/admin/**` 进入用户治理、内容治理、申诉治理与概览看板；举报治理通过 `/api/reports/admin*` 提供管理员能力。

3. 权限边界与交互策略

- 管理员治理接口以 `/api/admin/**` 为主，举报治理接口位于 `/api/reports/admin*`，并由后端管理员校验控制。
- 前端登录后，管理员会被引导至管理员中心页面；在当前实现中，管理员入口聚焦于管理能力，普通用户页面能力被弱化展示。

4. 设计意义

管理员角色通过“账号治理 + 内容审核 + 运行监控”三位一体机制，保证社区秩序、降低违规传播概率，并为系统可持续运营提供制度化支撑。

### 3.2.3 游客

游客用于降低首次使用门槛，支持“先体验、后注册”的轻量访问模式。

1. 角色定位

游客通过“游客登录”获得临时身份（ROLE_GUEST），可在不注册正式账号的前提下进入社区浏览。

2. 主要权限

- 可访问社区只读能力：`/api/community/feed`、`/api/community/recommendations`、`/api/community/hot-pets`、`/api/community/topics`、`/api/community/posts/{postId}/comments`。
- 可进入社区页面进行浏览与检索。

3. 权限边界

- 无法访问宠物主页管理、宠物详情维护、回忆编辑、个人资料编辑等私有功能。
- 无法执行社区写操作，如发帖、评论、点赞、创建话题、关注、私信、举报等。
- 无法访问 `/api/community/following-feed` 等需登录用户关系数据的接口。
- 无法访问管理员页面与管理员接口。

4. 设计意义

游客角色在不牺牲核心数据安全的前提下提升系统可达性：用户可先体验社区内容与产品价值，再决定是否注册成为普通用户，从而优化整体转化路径。

### 3.2.4 接口级权限映射（与实现一致）

1. 匿名开放接口

- `/api/auth/**`、`/api/public/**`、`/api/test/**`。
- `POST /api/account-appeals/public`（冻结用户公共申诉入口）。
- `GET /uploads/**`（上传文件静态访问）。

2. 游客可访问接口（ROLE_GUEST）

- 仅社区只读接口：`/api/community/feed`、`/api/community/recommendations`、`/api/community/hot-pets`、`/api/community/topics`、`/api/community/posts/{postId}/comments`。

3. 普通用户/管理员可访问接口（ROLE_USER/ROLE_ADMIN）

- 全部业务接口 `/api/**`（管理员接口 `/api/admin/**` 除外）。
- 社区写接口、社交关系、私信、举报、宠物与回忆、宠物档案提醒等能力均在该范围内。

4. 管理员专属接口（ROLE_ADMIN）

- `/api/admin/**`：用户治理、帖子评论审核、申诉处理、系统概览。
- `/api/reports/admin`、`/api/reports/admin/{id}/handle`：举报工单查看与处理（服务层管理员校验）。

## 小结

本系统通过普通用户、系统管理员、游客三类角色分工，形成了“业务生产者—平台治理者—体验观察者”的完整权限结构。该结构在技术实现上具备清晰的边界与可扩展性，在论文层面也能够支撑后续对安全性、可用性与运营机制的分析。
