## 3.3 功能性需求

本节在 3.2 用户角色分析基础上，按“普通用户—系统管理员—游客”三类参与者展开功能性需求。用例命名均采用动宾短语，且从用户视角描述系统可感知的业务功能。

### 3.3.1 用例建模范围与约定

1. 系统边界

本系统边界定义为“宠物数字纪念册 Web 系统”，包括账号认证、宠物与回忆管理、宠物档案提醒、社区互动、后台治理等业务子系统。

2. 参与者定义

- 普通用户：注册并登录的正式用户，承担内容生产与互动。
- 系统管理员：具备后台治理权限的管理者。
- 游客：通过游客登录获取只读访问能力的轻量用户。

3. 用例图放置方式

- 每类角色单独绘制一张用例图。
- 用例图放在对应角色小节末尾。
- 论文中可使用“用例图标题标记”作为插图定位锚点。

4. 接口权限映射约定

- 访客可访问范围以接口白名单为准：`/api/community/feed`、`/api/community/recommendations`、`/api/community/hot-pets`、`/api/community/topics`、`/api/community/posts/{postId}/comments`。
- 管理员治理能力主要通过 `/api/admin/**` 提供，并通过 `/api/reports/admin*` 处理举报工单，不与普通业务写接口混用。

### 3.3.2 普通用户功能需求

普通用户是系统核心角色，其功能覆盖“记录—分享—互动—沉淀”完整链路。

#### 普通用户用例描述（按标准模板）

##### UC-U01 注册账号

用例名称：注册账号用例编号：UC-U01参与者：普通用户用例描述：未登录用户填写用户名、邮箱和密码，创建正式账号并建立登录态。触发器：用户在注册页面点击“注册”。前置条件：用户未登录。后置条件：创建用户并返回登录态。基本事件流：

1. 用户进入注册页并填写用户名、邮箱、密码。
2. 系统校验必填项、邮箱格式和密码规则。
3. 系统校验用户名和邮箱唯一性。
4. 校验通过后系统创建账号并加密保存密码。
5. 系统返回登录令牌与用户信息。扩展事件流：
6. 用户名或邮箱已存在：系统提示冲突并阻止提交。
7. 邮箱格式非法或密码不符合规则：系统提示修正。
8. 数据保存失败：系统提示稍后重试。
   结论：成功时用户进入登录状态；失败时不创建账号。
   业务规则：用户名与邮箱全局唯一；邮箱必须通过格式校验；密码必须按安全策略加密存储。

##### UC-U02 登录系统

用例名称：登录系统用例编号：UC-U02参与者：普通用户用例描述：已注册用户通过用户名和密码登录系统，系统签发 JWT。触发器：用户在登录页面点击“登录”。前置条件：用户已注册。后置条件：登录成功并进入主页。基本事件流：

1. 用户输入用户名和密码并提交。
2. 系统查询账号并检查账号状态。
3. 系统校验密码哈希。
4. 校验通过后签发 JWT 并返回用户角色信息。
5. 前端按角色跳转到对应首页。扩展事件流：
6. 账户不存在：系统提示账户不存在。
7. 密码错误：系统提示密码错误。
8. 账号被冻结：系统拒绝登录并提示联系管理员。
   结论：成功时建立会话；失败时保持未登录状态。
   业务规则：冻结账号禁止登录；登录失败提示需明确区分原因；受保护接口仅接受有效 JWT。

##### UC-U03 完善个人资料

用例名称：完善个人资料用例编号：UC-U03参与者：普通用户用例描述：用户在个人中心维护昵称、头像和简介等个人信息。触发器：用户在个人中心点击“保存资料”。前置条件：用户已登录。后置条件：个人资料更新成功。基本事件流：

1. 用户打开个人中心查看当前资料。
2. 用户修改昵称、头像、简介等字段。
3. 系统进行字段格式与长度校验。
4. 校验通过后保存用户资料。
5. 系统返回最新资料并提示保存成功。扩展事件流：
6. 字段超长：系统提示超出长度限制。
7. 头像地址或简介格式不合法：系统提示修正。
8. 保存失败：系统提示重试。
   结论：成功时资料被更新；失败时保留原资料。
   业务规则：昵称、简介需满足长度边界；头像字段需符合 URL/上传返回值规则。

##### UC-U04 管理宠物信息

用例名称：管理宠物信息用例编号：UC-U04参与者：普通用户用例描述：用户新增或编辑宠物基础档案，包括公开状态配置。触发器：用户在宠物管理页点击“新增宠物”或“保存宠物”。前置条件：用户已登录。后置条件：宠物档案创建或更新。基本事件流：

1. 用户进入宠物管理页面并选择新增或编辑。
2. 用户填写昵称、物种、品种、性别、生日、纪念日、公开状态。
3. 系统校验必填项和字段合法性。
4. 校验通过后系统保存宠物信息。
5. 系统返回宠物详情并刷新列表。扩展事件流：
6. 必填项缺失：系统提示补全并拒绝保存。
7. 日期或枚举值非法：系统提示字段错误。
8. 宠物不存在或无权限：系统拒绝修改。
   结论：成功时宠物信息可被后续业务引用；失败时不产生变更。
   业务规则：仅宠物所属用户可编辑；宠物公开状态决定外部可见性。

##### UC-U05 记录宠物回忆

用例名称：记录宠物回忆用例编号：UC-U05参与者：普通用户用例描述：用户为指定宠物新增回忆条目并沉淀到时间线。触发器：用户在宠物详情页点击“新建回忆”。前置条件：用户已登录且已选中宠物。后置条件：回忆进入时间线展示。基本事件流：

1. 用户选择目标宠物并打开回忆编辑器。
2. 用户填写标题、内容、日期、地点并可附加媒体。
3. 系统校验标题和内容等必填字段。
4. 校验通过后系统保存回忆记录。
5. 时间线刷新并展示新回忆。扩展事件流：
6. 标题或内容为空：系统提示补全。
7. 目标宠物不存在或无权限：系统拒绝保存。
8. 保存失败：系统提示稍后重试。
   结论：成功时形成可追溯回忆记录；失败时不入库。
   业务规则：回忆必须归属某只宠物；回忆时间应采用统一时区与日期格式。

##### UC-U06 上传媒体文件

用例名称：上传媒体文件用例编号：UC-U06参与者：普通用户用例描述：用户上传图片或视频，系统返回可引用的资源 URL。触发器：用户在表单中点击“上传文件”。前置条件：用户已登录。后置条件：媒体资源可用于业务展示。基本事件流：

1. 用户选择本地图片或视频文件。
2. 系统校验文件类型和大小。
3. 校验通过后系统上传文件并生成访问 URL。
4. 前端回填 URL 到当前业务表单。
5. 用户继续完成业务提交。扩展事件流：
6. 文件类型不支持：系统拒绝上传并提示支持格式。
7. 文件大小超限：系统提示超出上限。
8. 存储服务异常：系统提示上传失败。
   结论：成功时文件可被业务页面引用；失败时表单不回填媒体地址。
   业务规则：仅允许白名单媒体类型；单文件大小受平台上限约束。

##### UC-U07 维护宠物档案

用例名称：维护宠物档案用例编号：UC-U07参与者：普通用户用例描述：用户维护宠物健康档案记录，支持新增、编辑、删除。触发器：用户在宠物详情页点击“新增档案”或“保存档案”。前置条件：用户已登录且已进入宠物详情。后置条件：宠物大事记可追溯。基本事件流：

1. 用户选择档案类型（体重、疫苗、体检、驱虫、用药等）。
2. 用户填写事项名称、日期、备注及提醒配置。
3. 系统校验名称和类型合法性。
4. 系统保存档案记录并更新列表。
5. 用户可对既有记录继续编辑或删除。扩展事件流：
6. 事项名称为空：系统拒绝保存并提示。
7. 事项类型非法：系统拒绝保存。
8. 无权限操作他人宠物档案：系统拒绝请求。
   结论：成功时档案形成连续记录；失败时档案不变更。
   业务规则：档案类型需在预定义枚举内；仅宠物所属用户可维护。

##### UC-U08 设置健康提醒

用例名称：设置健康提醒用例编号：UC-U08参与者：普通用户用例描述：用户为档案记录开启提醒并设置提醒时间。触发器：用户在档案编辑页启用“提醒开关”并点击保存。前置条件：用户已登录且新增/编辑宠物档案。后置条件：到期提醒可在登录后弹窗触发。基本事件流：

1. 用户在档案记录中开启提醒。
2. 用户填写提醒时间与说明。
3. 系统校验提醒时间格式与有效性。
4. 校验通过后系统保存提醒计划。
5. 到期时系统在用户登录后展示提醒弹窗。扩展事件流：
6. 开启提醒但未填写时间：系统提示补全。
7. 提醒时间非法或早于当前时间：系统拒绝保存。
8. 保存失败：系统提示重试。
   结论：成功时提醒进入待触发状态；失败时不创建提醒。
   业务规则：提醒开关为开启时，提醒时间必填；提醒时间需满足系统有效时间规则。

##### UC-U09 处理提醒事项

用例名称：处理提醒事项用例编号：UC-U09参与者：普通用户用例描述：用户对已到期提醒执行“已完成”或“稍后处理”。触发器：系统弹出到期提醒，用户点击处理按钮。前置条件：用户已登录且存在到期提醒。后置条件：提醒状态被更新或顺延。基本事件流：

1. 系统展示到期提醒弹窗与事项详情。
2. 用户选择“已完成”或“稍后”。
3. 若选择已完成，系统将提醒状态更新为完成。
4. 若选择稍后，用户输入新的提醒时间。
5. 系统保存处理结果并刷新提醒列表。扩展事件流：
6. 稍后时间非法：系统拒绝并提示修正。
7. 提醒已被其他终端处理：系统提示状态已更新。
8. 保存失败：系统提示稍后重试。
   结论：成功时提醒形成闭环处理记录；失败时维持原状态。
   业务规则：完成状态不可回退；顺延时间必须晚于当前时间。

##### UC-U10 浏览社区动态

用例名称：浏览社区动态用例编号：UC-U10参与者：普通用户用例描述：用户浏览社区信息流并按条件筛选帖子。触发器：用户进入社区页面或提交筛选条件。前置条件：用户已登录。后置条件：展示筛选后的帖子列表。基本事件流：

1. 用户进入社区页面。
2. 系统加载默认帖子列表。
3. 用户设置叙事时态、情绪标签、关键词等筛选项。
4. 系统按条件查询并返回结果。
5. 页面展示帖子卡片与分页/加载更多结果。扩展事件流：
6. 无匹配结果：系统显示空态。
7. 查询失败：系统提示重试。
8. 条件无效：系统恢复默认筛选。
   结论：成功时用户获得目标内容列表；失败时保留当前可见数据。
   业务规则：仅展示用户有权限查看的帖子；筛选条件采用统一查询参数规范。

##### UC-U11 发布社区动态

用例名称：发布社区动态用例编号：UC-U11参与者：普通用户用例描述：用户发布帖子到社区信息流。触发器：用户在社区页点击“发布”并提交表单。前置条件：用户已登录且账号可发布。后置条件：新帖子进入社区信息流。基本事件流：

1. 用户打开发布弹窗并填写标题、正文、媒体、话题。
2. 系统校验字段完整性与长度规则。
3. 校验通过后系统保存帖子并建立作者关联。
4. 系统返回发布成功结果。
5. 信息流刷新，展示新帖子。扩展事件流：
6. 用户被限制发布：系统拒绝发布并提示。
7. 字段校验失败：系统提示修正后重试。
8. 发布失败：系统提示稍后重试。
   结论：成功时帖子可被他人浏览互动；失败时不写入帖子。
   业务规则：限制发布状态下禁止新增帖子；标题与正文需满足最小/最大长度规则。

##### UC-U12 互动社区内容

用例名称：互动社区内容用例编号：UC-U12参与者：普通用户用例描述：用户对帖子执行点赞、评论和纪念接力等互动行为。触发器：用户在帖子卡片点击点赞、评论或接力按钮。前置条件：用户已登录且账号可发布。后置条件：帖子互动数据更新。基本事件流：

1. 用户选择目标帖子并执行互动操作。
2. 系统校验目标帖子存在且可互动。
3. 系统写入互动记录并更新统计计数。
4. 系统返回最新互动状态。
5. 页面实时更新互动数与用户操作态。扩展事件流：
6. 目标不存在：系统提示帖子不可用。
7. 权限不足或账号受限：系统拒绝操作。
8. 评论内容为空：系统提示补全后提交。
   结论：成功时互动行为被记录；失败时不产生互动变更。
   业务规则：同一用户对同一帖子点赞应幂等；评论内容不得为空。

##### UC-U13 维护社交关系

用例名称：维护社交关系用例编号：UC-U13参与者：普通用户用例描述：用户关注或取消关注其他用户，并查看关注动态。触发器：用户在他人主页点击“关注/取消关注”。前置条件：用户已登录。后置条件：关注关系与关注流更新。基本事件流：

1. 用户访问目标用户主页。
2. 用户执行关注或取消关注操作。
3. 系统校验目标用户有效性与关系状态。
4. 系统更新关注关系数据。
5. 关注动态页面按新关系刷新内容。扩展事件流：
6. 关注目标无效：系统提示目标不存在。
7. 重复关注/重复取关：系统按幂等规则返回提示。
8. 试图关注自己：系统拒绝操作。
   结论：成功时关系状态与关注流一致；失败时关系不变。
   业务规则：用户不能关注自己；同一关注关系在系统中唯一。

##### UC-U14 发送私信

用例名称：发送私信用例编号：UC-U14参与者：普通用户用例描述：用户在私信会话中向目标用户发送文本消息。触发器：用户在私信窗口点击“发送”。前置条件：用户已登录。后置条件：会话记录写入并可查询。基本事件流：

1. 用户选择会话对象并打开聊天窗口。
2. 用户输入消息内容并点击发送。
3. 系统校验接收方存在与内容合法性。
4. 系统写入消息记录并返回发送结果。
5. 会话列表与消息窗口展示新消息。扩展事件流：
6. 接收方不存在：系统提示发送失败。
7. 内容为空：系统拒绝发送并提示补全。
8. 写入失败：系统提示重试。
   结论：成功时消息可在会话中追溯；失败时不写入消息。
   业务规则：消息内容不能为空并受长度限制；仅登录用户可发送私信。

##### UC-U15 举报违规内容

用例名称：举报违规内容用例编号：UC-U15参与者：普通用户用例描述：用户提交违规内容举报单，进入管理员处理队列。触发器：用户在内容页面点击“举报”并提交。前置条件：用户已登录。后置条件：举报单进入管理员处理队列。基本事件流：

1. 用户选择举报目标（帖子/评论/用户）。
2. 用户填写举报理由与补充说明。
3. 系统校验目标有效性与理由完整性。
4. 校验通过后系统创建举报工单。
5. 系统返回提交成功并提示等待处理。扩展事件流：
6. 理由为空：系统拒绝受理并提示补全。
7. 目标无效或已删除：系统提示目标不可举报。
8. 提交失败：系统提示稍后重试。
   结论：成功时举报形成待处理工单；失败时不产生工单。
   业务规则：举报理由必填；举报目标必须为系统内有效对象。

##### UC-U16 分享公开主页

用例名称：分享公开主页用例编号：UC-U16参与者：普通用户用例描述：用户生成公开主页链接并分享给外部访问者。触发器：用户在主页设置页点击“生成分享链接”。前置条件：用户已登录且至少有一只公开宠物。后置条件：外部访问者可访问公开主页。基本事件流：

1. 用户进入公开主页配置页面。
2. 系统检查用户是否存在公开宠物。
3. 系统生成包含 ownerUsername 与 pet 参数的链接。
4. 用户复制链接并分享。
5. 访问者通过链接查看公开主页内容。扩展事件流：
6. 无公开宠物：系统提示先公开至少一只宠物。
7. 链接生成失败：系统提示稍后重试。
8. 目标宠物被设为私有：外部访问时仅展示可公开内容。
   结论：成功时对外分享路径可用；失败时不产生有效分享链接。

后置条件：评论列表成功展示或返回对应提示。
基本事件流：

1. 游客选择目标帖子并打开详情页。
2. 系统校验帖子可见性并加载评论列表。
3. 系统按时间或热度返回评论数据。
4. 页面渲染评论内容与基础互动统计。
5. 游客浏览评论并返回列表页。
   扩展事件流：
6. 目标帖子不存在或已下线：系统提示内容不可用。
7. 评论数据加载失败：系统提示稍后重试。
8. 评论区被关闭：系统提示暂不支持查看评论。
   结论：成功时游客获得互动上下文信息；失败时维持当前会话不变。
   业务规则：游客仅可查看评论，不可发表评论；仅公开帖子及其可见评论可被读取。

##### UC-G05 升级正式账号

用例名称：升级正式账号
用例编号：UC-G05
参与者：游客
用例描述：游客通过注册或登录流程升级为正式账号，获取完整业务能力。
触发器：游客点击“注册/登录以解锁更多功能”。
前置条件：游客已登录。
后置条件：角色从游客升级为普通用户并建立正式会话。
基本事件流：

1. 游客在受限功能处点击升级入口。
2. 系统跳转到注册或登录页面。
3. 用户填写注册信息或输入已有账号凭据。
4. 系统完成账号校验并签发正式用户令牌。
5. 前端替换游客会话并跳转普通用户首页。
   扩展事件流：
6. 注册信息冲突：系统提示用户名或邮箱已存在。
7. 输入信息不合法：系统提示修正后重试。
8. 用户中途取消：系统保留游客会话并返回原页面。
   结论：成功时用户完成角色升级；失败时维持游客身份不变。
   业务规则：升级成功后应失效旧游客令牌；升级过程需遵循普通用户注册与登录安全规则；角色变更需即时生效。

### 3.3.5 功能需求小结

本系统功能需求通过三类角色分层实现：普通用户承担业务内容生产，系统管理员承担秩序治理与运维监控，游客承担低门槛体验入口。用例集合既覆盖核心业务闭环，也覆盖治理闭环与体验闭环，具备较完整的毕业设计可实现性。

## 3.4 非功能性需求

非功能性需求用于约束系统“怎么做得好”，重点覆盖安全性、性能、可用性、可靠性、可维护性与可扩展性。以下需求均给出“设置原因”与“本系统解决办法”。

### 3.4.2 重点非功能需求说明

1. 安全性

本系统采用“后端强校验、前端弱引导”的组合策略。后端负责最终权限判定，避免仅依赖前端拦截；前端负责提升交互一致性，减少误操作。

2. 性能与体验平衡

系统将高频访问场景（社区信息流、评论、时间线）作为性能优化重点。通过索引设计与按需加载，保证在功能丰富的同时维持可接受交互延迟。

3. 可靠性与治理可追溯

管理员对用户、内容、举报、申诉的处理均以状态化方式流转，结合时间戳字段，可形成可追溯治理链路，便于论文后续“系统运行管理”章节展开。

### 3.4.3 非功能需求小结

非功能需求并非附加条件，而是系统可上线、可长期运行的基础约束。通过将“原因—方案—落地机制”三者对应，本系统在毕业设计范围内实现了较完整的工程化质量保障。
